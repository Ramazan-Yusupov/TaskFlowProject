# Правило сохранения пользовательского дизайна

Пользователь редактирует визуальную часть TaskFlow самостоятельно. Эти изменения считаются источником истины.

## Нельзя без отдельной явной задачи

- менять `src/index.css`, Tailwind-классы, цвета, типографику, отступы, сетки, брейкпоинты и существующую структуру layout-компонентов;
- откатывать или «улучшать» визуальные правки пользователя во время функциональной задачи;
- заменять иконки, декоративные элементы или тексты в уже отредактированном UI;
- массово форматировать файлы с визуальной разметкой, если это может затронуть дизайн.

## Можно

- добавлять доменную логику, API, тесты, persistence, валидацию и обработку ошибок;
- подключать функциональность к существующим элементам управления;
- создавать новый изолированный UI только когда без него нельзя выполнить пользовательский сценарий, используя уже существующие визуальные primitives и не меняя текущие экраны.

## Рабочий порядок

1. Перед задачей зафиксировать изменённые пользователем стили и layout-файлы.
2. Вносить изменения в model, API, backend и отдельные feature-модули.
3. Перед выдачей архива проверить, что зафиксированные дизайн-файлы не изменены.
4. В отчёте явно указать, какие design-файлы намеренно не затрагивались.
