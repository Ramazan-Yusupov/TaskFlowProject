# План развития TaskFlow

Этот roadmap описывает последовательность функциональных изменений. Визуальный дизайн, изменяемый пользователем, не является частью этих этапов и защищён правилами из `docs/DESIGN_PRESERVATION.md`.

## Этап 0 — защита дизайна · выполнен

- Зафиксировано правило: визуальные правки пользователя не изменяются без отдельной дизайн-задачи.
- Функциональные доработки выполняются в domain, application, infrastructure, API gateway и изолированных feature-модулях.
- Перед выдачей каждого архива проверяется неизменность зафиксированных дизайн-файлов.

## Этап 1 — жизненный цикл задач · выполнен

- Создание, просмотр и редактирование задачи через существующие сценарии интерфейса.
- Удаление задачи с явным подтверждением.
- Локальное сохранение demo-данных в браузере.
- Валидируемая смена статуса только на следующий этап потока.

## Этап 2 — надёжный backend-контур · выполнен

- CRUD API для задач.
- Файловое persistence-хранилище для локальной разработки.
- Ограничение размера HTTP body и единый формат ошибок.
- Unit-тесты use case для создания, обновления, удаления и перехода статуса.

## Этап 3 — пользователи и рабочие пространства · следующий

1. Ввести `User`, `Workspace`, `Membership`, роли `owner`, `manager`, `member`.
2. Добавить регистрацию, вход, refresh-token и server-side authorization checks.
3. Привязать каждую задачу к рабочему пространству и проверять доступ на каждом запросе.
4. Заменить файловый adapter на PostgreSQL adapter, не меняя application use cases.

## Этап 4 — командная работа

1. Добавить комментарии, историю изменений и счётчики непрочитанного.
2. Добавить назначение исполнителя, метки, проекты и расширенные серверные фильтры.
3. Подключить уведомления через outbox/event layer и WebSocket.

## Этап 5 — production readiness

1. Миграции, Docker Compose, health/readiness checks и конфигурация окружений.
2. Rate limiting, security headers, structured logs, request ID и аудит действий.
3. E2E-покрытие критичных сценариев и CI pipeline.
